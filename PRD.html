Product Requirements Document

VR 3D Coloring Game Demo — powered by Funes.World models

Version: 0.4 Last updated: 2026-06-08 Status: Draft / Concept Owner: SpiralArmZ

1. Vision & Purpose

A VR puzzle game demo with a soothing, cozy vibe. The player picks up a white (uncolored) photoscanned architectural model and uses paint and spray tools to slowly reveal its full color — a tactile, meditative "3D coloring book." The aim of this demo is to validate the core reveal loop and the calming tone, not to ship a full game.

2. Goals & Non-Goals

Goals (demo)

  • One fully playable Funes.World model that can be colored start to finish.
  • A satisfying, tactile color-reveal loop using spray + brush tools.
  • A calm, cozy atmosphere (audio, lighting, pacing, haptics).
  • Runs comfortably on Meta Quest 3 / 3S standalone at a stable framerate.

Non-Goals (demo)

  • No multiplayer.
  • No meta-progression, economy, or unlock systems.
  • No in-game level/model editor.
  • No fail states, scoring pressure, or timers.

5. Gameplay & Mechanics

Primary mechanic (MVP) Selected

Spray + brush reveal. A spray can flood-reveals broad color zones of the model; a brush refines detail and edges. This is the most tactile and "cozy" option and is closest in feel to Powerwash Simulator. The demo is built around this loop.

Alternative mechanics (documented, not built for the demo)

  • Color-sequence guide — follow a guided order of zones/colors. Structured and low-frustration, but risks monotony.
  • X-Ray then paint — use a scan/X-ray tool to reveal a "where to paint" map first, then color. More puzzle-forward.
  • Pick-and-place elements — extract colored elements and place them onto the white model to reveal areas step by step. Most novel, highest build cost.

These remain candidates for layering on top of the MVP loop if the core feel needs more depth.

Layering strategy: ship spray + brush first. If playtests show the loop needs more depth, the most natural additions are an optional X-ray "where to paint" hint or a light sequence/goal overlay — added without removing the freeform, no-pressure core.

6. Core Loop & UX

  1. Grab the white model and bring it to a comfortable inspection distance.
  2. Inspect — turn it over, find areas to color.
  3. Spray broad color zones with the spray tool.
  4. Brush in detail and edges.
  5. Completion beat — a gentle, satisfying payoff when the model is fully revealed.

Design pillars: soothing audio, gentle haptics tied to tool contact, no fail state, no timer, and readable, diegetic tools (the can and brush are physical objects, minimal floating UI).

7. Content & Assets

Models are photoscanned architectural assets from Funes.World, used with the founder's permission for public and commercial settings (a personal relationship with the founder).

License caveat: "free" 3D assets are typically published under a specific Creative Commons license — CC BY allows commercial use with attribution, CC0 waives attribution, but CC BY-NC forbids commercial use and CC BY-ND forbids modification/recoloring. Since this project recolors models commercially, the grant should explicitly cover commercial use AND derivatives. Getting this in writing is an open item (see §10).

8. Technical Requirements

  • Engine: Unity 6 (editor 6000.3.17f1), URP (mobile renderer for Quest).
  • VR SDK: Meta XR All-in-One SDK (incl. Interaction SDK for grab + tool interactions), via OpenXR + the Meta Quest feature group.
  • Platform build: Android / Quest standalone; mobile-class performance budget.
  • Reveal technique (to prototype): texture-space painting (paint into a render texture / mask), vertex-color painting, or a shader-driven reveal mask. Trade-offs in §11. Keep it WebGL2-safe if the web stretch goal matters: WebGL2 has no compute shaders, so a compute-based painter would not port. Fragment / render-texture painting and vertex-color approaches run on WebGL2 cleanly.
  • Performance: hold target framerate on Quest 3/3S; budget poly count and texture memory of photoscanned assets accordingly (decimation / texture compression likely required).

Architecture — decouple player actions from Meta input

Route every player action (paint/spray: hit point, target surface, brush radius, color, trigger strength) through an input-agnostic interface. The Meta Interaction SDK (controllers / hands) is treated as just one input driver behind that interface — not wired directly into the paint/reveal logic. This keeps the gameplay portable: a non-VR touch/mouse driver can feed the same actions for a mobile-browser (WebGL) build, and other XR runtimes (Pico, Android XR) can plug in similarly.

Corollary: the cozy core scene and reveal logic must be buildable without Meta packages (Meta XR SDK is excluded from any WebGL build). This is nearly free to honor now and expensive to retrofit later, so it is a standing design constraint from the first prototype.

9. Success Criteria (demo)

  • A first-time player understands and completes the color-reveal loop without instruction.
  • Testers describe the experience as calming/satisfying.
  • Stable framerate on Quest 3/3S throughout a full model.
  • The reveal interaction feels physical and responsive (good haptics + visual feedback).

10. Risks & Open Questions

  • Reveal tech: which painting approach gives the best feel within the Quest perf budget?
  • Asset budget: photoscanned models are heavy — how much decimation/retexturing is needed?
  • Depth vs. boredom: is the spray+brush loop engaging enough alone, or does it need a layered mechanic?
  • Web portability: does the chosen reveal shader stay WebGL2-safe (no compute), and is the input layer abstracted enough that a touch driver "just works"? Decided early or it gets costly.
  • Licensing: formal wording for Funes.World asset reuse in this repo.

11. Roadmap

Dates intentionally omitted while in the concept stage.

Milestone 0 — Foundations (current)

  • Create repository and the living PRD. (done)
  • Scaffold the Unity 6 URP project at the repo root; editor updated to 6000.3.17f1. (done)
  • Confirm asset-license wording with Funes.World (in writing). (open)

Milestone 1 — Reveal prototype

  • Install Meta XR All-in-One SDK; switch to Android; run Meta Project Setup Tool. (next)
  • Grab a model and hold a spray can / brush (Interaction SDK).
  • Prototype 1–2 reveal techniques (texture-space and/or vertex-color); compare feel + perf; pick one.

Milestone 2 — One cozy model, end to end

  • Import a Funes.World model; decimate/retopo and compress textures for Quest.
  • Full color-reveal of a single model with spray + brush.
  • Cozy layer: ambient audio, lighting, gentle haptics, completion beat.

Milestone 3 — Playable demo build

  • Stable framerate on Quest 3/3S across a full model.
  • First-time players complete the loop without instruction.
  • Internal playtest: "is it calming/satisfying?"

Milestone 4 — Evaluate stretch goals

  • Assess PCVR / Pico 4 / Android XR / Apple Vision Pro via OpenXR.
  • Spike a non-VR Unity WebGL build of a tiny level with a touch input driver (mobile-browser goal). Relies on the §8 input-decoupling holding up. Add the Web module to the editor only at this point.
  • Research AR-glasses feasibility (XReal, Spectacles).
